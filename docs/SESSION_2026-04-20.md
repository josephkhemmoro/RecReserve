## 2. Stripe Payment Error — Fixed

**Symptom:** Booking flow on TestFlight kept throwing "You did not provide an API key. You need to provide your API key in the Authorization header…"

**Root causes (layered):**
1. The secret key on Supabase and the publishable key baked into the TestFlight build were from **two different Stripe accounts** (pk prefix `TFSTeLCXhkA5ESL` vs. sk prefix `TFSTWLcjAPX63wH`)
2. When switching accounts, initially set a test secret key while the mobile app had a live publishable key (mode mismatch)

**Fix:**
- Set matching `sk_live_...TFSTWLcjAPX63wH` on Supabase via `supabase secrets set STRIPE_SECRET_KEY=...`
- Updated mobile `.env` and `eas.json` production env block with matching `pk_live_...TFSTWLcjAPX63wH`
- Redeployed `create-payment-intent`, `create-event-payment`, `process-refund`, `finalize-booking`

**Result:** Payment Sheet opens cleanly. First live $18 transaction went through.

**Takeaway for future:** always verify the 15-char account ID portion after `pk_test_51` / `sk_test_51` (or `_live_51`) matches between publishable and secret keys. Mismatch = "did not provide API key".

## 3. Zustand Persistence / Cache

**Symptom:** Every reload flashed the "Join a club" empty state for 1–3 seconds before the real club loaded.

**Root cause:** `clubStore` and `membershipStore` were in-memory only, so `selectedClub` was `null` on every cold start until `checkMemberships` finished fetching from Supabase.

**Fix:** Wrapped both stores in Zustand's `persist` middleware backed by AsyncStorage. Partialized to essential fields only (loading/error states still start fresh). Existing `clearClub()` / `clearMembership()` calls on sign-out continue to work because `persist` writes on every `set`.

**Files:**
- `apps/mobile/store/clubStore.js`
- `apps/mobile/store/membershipStore.js`

**Result:** App shows the correct club instantly on launch. Background refetch still keeps data current (stale-while-revalidate pattern).

## 4. Paid Memberships — Full Feature

Players can now upgrade/downgrade memberships, and clubs can require a paid tier at join time.

### Migration — `supabase/migrations/20260420_paid_memberships.sql`

## 5. PGRST201 Ambiguity Fix

**Symptom:** Mobile threw `PGRST201` "more than one relationship found" when fetching membership + tier.

**Root cause:** The paid-memberships migration added `pending_tier_id UUID REFERENCES membership_tiers(id)` — a second FK between `memberships` and `membership_tiers`.

**Fix:** Appended `!tier_id` to every embed that traverses memberships → membership_tiers to disambiguate.

## 8. Streak Milestone Rewards — Full Gamification Feature

The "make clubs stickier" play. Players who keep coming back earn discount / free-booking rewards from their clubs.

### Migration — `supabase/migrations/20260420_streak_rewards.sql`

- **`milestone_rewards`** — club-editable templates (milestone int, reward_type enum, reward_value, title, description, expires_days, is_active)
- **`player_rewards`** — per-player grants with snapshot fields (so deleting the template doesn't wipe earned rewards). Tracks `granted_at`, `expires_at`, `redeemed_at`, `redeemed_reservation_id`
- **DB trigger** on `streak_milestones` insert → auto-grants all matching active `milestone_rewards` to the player
- **RPC `redeem_player_reward(reward_id, user_id, reservation_id)`** — atomic lock + validate + mark redeemed

Reward types:
- `discount_percent` — N% off next booking
- `free_booking` — free court
- `bonus_credit` — adds cents to user's credit_balance (admin can create, not yet spendable — v1 stub)

### Admin

New page `/rewards-program` under Engagement, with 5 preset milestone sections (4/8/12/26/52 weeks) + custom milestones row. Each reward card shows earned/redeemed counts. Top-of-page adoption stats.

### Mobile

- New `store/rewardsStore.js` — persisted, with available/redeemed/expired getters
- New screen `app/rewards/index.jsx` with tabs + countdown badges (red if ≤3 days)
- Entry points: profile list row with count badge, StreakStrip pill
- Booking confirm (`app/booking/confirm.jsx`) has new "Apply a reward" card — dynamic price recompute, free-booking case skips PaymentSheet entirely, atomic redemption via RPC after reservation insert

### `create-payment-intent` — updated

Accepts optional `reward_id`, validates ownership + unredeemed + unexpired, applies discount server-side. Returns `reward_applied`, `discounted_amount`, `reward_skipped` flags. For free_booking → returns `clientSecret: null, final_amount: 0` so mobile skips PaymentSheet. Backward compatible: calls without `reward_id` behave exactly as before.

### Testing tool

SQL to simulate a 4-week milestone for visual testing:

```sql
INSERT INTO player_streaks (user_id, club_id, current_streak, longest_streak, last_play_week)
VALUES ('USER_UUID', 'CLUB_UUID', 4, 4, date_trunc('week', now())::date)
ON CONFLICT (user_id, club_id) DO UPDATE SET
  current_streak = 4,
  longest_streak = GREATEST(player_streaks.longest_streak, 4),
  last_play_week = EXCLUDED.last_play_week,
  updated_at = now();

INSERT INTO streak_milestones (user_id, club_id, milestone)
VALUES ('USER_UUID', 'CLUB_UUID', 4)
ON CONFLICT (user_id, club_id, milestone) DO NOTHING;
```

DB trigger auto-grants any active matching rewards.
