## 3. Booking Flow — Atomic Path Wired Up

`apps/mobile/app/booking/confirm.jsx`:
- Switched from direct `reservations.insert` to invoking the deployed `finalize-booking` edge function.
- `finalize-booking` re-validates server-side, checks idempotency (so retries don't double-book or double-charge), verifies the Stripe PI is `succeeded` before writing, inserts the reservation + `payment_records` + audit log + push notification atomically.
- **Important error-handling distinction:** structured validation/payment errors from `finalize-booking` surface to the user immediately (no fallback). Only infrastructure failures (function unreachable / network) trigger the legacy direct-insert fallback — that's intentional, because the user has already been charged at that point and we want the reservation row to exist.
